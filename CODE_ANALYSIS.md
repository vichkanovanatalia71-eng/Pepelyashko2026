# Аналіз коду проекту Pepelyashko

**Дата:** 2026-02-23

## 1. Загальний огляд

**Pepelyashko** — повнофункціональний веб-застосунок для фінансового менеджменту медичних практик ФОП (фізичних осіб-підприємців) в Україні. Замінює Excel-ведення обліку автоматизованою системою для відстеження доходів, витрат та податків.

### Стек технологій

| Рівень     | Технологія                                  |
|------------|---------------------------------------------|
| Backend    | Python 3.12, FastAPI 0.115.6, SQLAlchemy 2.0, Alembic |
| Frontend   | React 18.3, TypeScript 5.6, Vite 6.0, Tailwind CSS |
| БД         | PostgreSQL 16 (async через asyncpg)         |
| AI         | Anthropic Claude API, OpenAI API            |
| Інфра      | Docker + docker-compose                     |

### Масштаб проекту

- **59** Python-файлів (backend)
- **41** TypeScript-файл (frontend)
- **14** моделей БД, **16** міграцій Alembic
- **15** модулів API-маршрутів, **17** фронтенд-сторінок

## 2. Архітектура — Сильні сторони

### 2.1 Чітке розділення шарів (Backend)

Бекенд має правильну шарову архітектуру:

```
app/
├── api/routes/    — HTTP-ендпоінти (контролери)
├── models/        — SQLAlchemy ORM-моделі
├── schemas/       — Pydantic-схеми валідації
├── core/          — Конфігурація, безпека, залежності
├── services/      — Бізнес-логіка
├── ai/            — AI-інтеграція
└── db/            — Сесія БД
```

Це добра організація для проекту такого розміру.

### 2.2 Повна асинхронність

Увесь бекенд побудований на `async/await` — від HTTP-обробників до роботи з БД (`asyncpg`). Це правильний підхід для FastAPI.

### 2.3 Безпечна аутентифікація

- JWT-токени з bcrypt-хешуванням паролів (`security.py:11-22`)
- Email-верифікація при реєстрації з fallback на auto-verify для dev-режиму (`auth.py:46-58`)
- OAuth2PasswordBearer з правильним dependency injection (`deps.py:9-42`)

### 2.4 Сучасний стек

- SQLAlchemy 2.0 з `Mapped[]` типізацією (замість старого `Column()`)
- Pydantic v2 для валідації
- React 18 з TypeScript та сучасним маршрутизатором
- Radix UI / shadcn/ui — зрілі UI-компоненти

### 2.5 Добрий Docker-конфіг

- Healthcheck для PostgreSQL перед запуском бекенду (`docker-compose.yml:13-16`)
- Volumes для персистентних даних
- Розділення сервісів (БД, бекенд, фронтенд)

## 3. Проблеми безпеки

### 3.1 КРИТИЧНО: CORS дозволяє всі домени

**Файл:** `backend/app/core/config.py:20`

```python
cors_origins: list[str] = ["*"]
```

Коментар каже "Railway frontend proxies via nginx", але `allow_credentials=True` з `allow_origins=["*"]` — небезпечна комбінація. Браузери заблокують credentials з wildcard origin, але це сигналізує про неправильну конфігурацію. Потрібно явно вказати дозволені домени для production.

**Рекомендація:** Використовувати змінну оточення для дозволених origin-ів:
```python
cors_origins: list[str] = ["http://localhost:5173"]  # override через env
```

### 3.2 КРИТИЧНО: Секретний ключ за замовчуванням

**Файл:** `backend/app/core/config.py:15`

```python
secret_key: str = "change-me-in-production"
```

А в `docker-compose.yml:25`:
```yaml
SECRET_KEY: dev-secret-key
```

Якщо хтось забуде змінити — всі JWT-токени будуть передбачуваними. Рекомендація: генерувати випадковий ключ при запуску або вимагати його наявності.

### 3.3 СЕРЕДНЄ: Відсутня валідація сили паролю

**Файл:** `backend/app/api/routes/auth.py:25`

При реєстрації (`UserCreate`) немає перевірки мінімальної довжини або складності паролю. Користувачі можуть встановити пароль "1".

### 3.4 СЕРЕДНЄ: Відсутній rate limiting

Ендпоінти `/api/auth/login` та `/api/auth/register` не мають обмеження частоти запитів. Це дозволяє brute-force атаки.

### 3.5 НИЗЬКЕ: Токен верифікації не має TTL

**Файл:** `backend/app/api/routes/auth.py:62-85`

`verification_token` не має часу дії. Старі токени залишаються валідними назавжди.

## 4. Архітектурні проблеми

### 4.1 Надмірно великі файли маршрутів (God Routes)

Деякі файли маршрутів мають занадто багато логіки:

| Файл                  | Розмір  | Проблема                                   |
|-----------------------|---------|--------------------------------------------|
| `monthly_expenses.py` | ~61.5 KB | Бізнес-логіка змішана з HTTP-обробниками  |
| `reports.py`          | ~50 KB  | Складні SQL-запити прямо в ендпоінтах     |
| `monthly_services.py` | ~40 KB  | Розрахунки в маршрутах замість сервісів    |
| `services.py`         | ~30 KB  | Аналогічно                                |

**Рекомендація:** Винести бізнес-логіку в окремий шар `services/` (наразі використовується лише `services/nhsu.py` та `services/ai_provider.py`). Маршрути повинні лише:
1. Приймати запит
2. Викликати сервіс
3. Повертати відповідь

### 4.2 Відсутній шар репозиторіїв

Усі SQL-запити виконуються безпосередньо в route-хендлерах. Це ускладнює:
- Повторне використання запитів
- Тестування (потрібно мокати всю БД замість одного шару)
- Підтримку та рефакторинг

### 4.3 Дублювання запитів до NhsuSettings

В багатьох файлах повторюється однаковий патерн:
```python
nhsu_result = await db.execute(
    select(NhsuSettings).where(NhsuSettings.user_id == user_id)
)
nhsu = nhsu_result.scalar_one_or_none()
```

Це зустрічається в `reports.py`, `monthly_expenses.py`, `taxes.py` тощо. Потрібна спільна утиліта.

### 4.4 Frontend: token зберігається в localStorage

**Файл:** `frontend/src/hooks/useAuth.tsx:16-18`, `frontend/src/api/client.ts:8`

`localStorage` вразливий до XSS-атак. HttpOnly cookie — безпечніший варіант для JWT-токенів. Це загальноприйнята практика для SPA, але варто зазначити як потенційне поліпшення.

## 5. Якість коду

### 5.1 Відсутні тести

**Це найбільша проблема проекту.** Немає жодного тесту:
- Ні юніт-тестів
- Ні інтеграційних тестів
- Ні E2E-тестів
- Немає конфігурації pytest, незважаючи на наявність pytest в залежностях

Для фінансового додатку, де правильність розрахунків критична (податки, ЄСВ, ВЗ), відсутність тестів — серйозний ризик.

### 5.2 Немає CI/CD

Відсутні GitHub Actions або будь-які інші CI/CD-конфігурації. Немає автоматичної перевірки:
- Лінтингу
- Типізації
- Тестів
- Збірки

### 5.3 Немає логування

Відсутнє структуроване логування. При помилках у production буде складно діагностувати проблеми. FastAPI підтримує middleware для логування запитів.

### 5.4 Немає обробки помилок на рівні застосунку

Відсутній глобальний обробник виключень (`exception_handler`). Необроблені помилки повернуть 500 з деталями реалізації.

## 6. Проблеми продуктивності

### 6.1 N+1 запити

В `monthly_expenses.py:58-77` (`_paid_services_income`) — спочатку запитуються всі звіти, потім окремий запит для записів. Аналогічний патерн у `reports.py`. При великій кількості записів це створить проблеми.

**Рекомендація:** Використовувати JOIN з агрегацією в одному SQL-запиті.

### 6.2 Відсутнє кешування

Звіти та дашборд виконують багато SQL-запитів при кожному завантаженні. Для даних, які не змінюються часто (місячні закриті звіти), варто додати кешування.

### 6.3 Відсутній пагінація

`list_incomes` та `list_expenses` (`incomes.py:48-64`) повертають усі записи за період без `limit/offset`. При великих обсягах даних це може бути проблемою.

## 7. Специфічні зауваження

### 7.1 Налаштування ЄСВ захардкожене

**Файл:** `backend/app/core/config.py:24`

```python
esv_monthly: float = 1760.00  # ЄСВ мінімальний (оновлюється щорічно)
```

Ставка ЄСВ залежить від мінімальної зарплати і змінюється щороку. Зараз вона частково настроюється через `NhsuSettings`, але дефолт жорстко прописаний.

### 7.2 Час дії JWT-токена — 8 годин

**Файл:** `backend/app/core/config.py:17`

```python
access_token_expire_minutes: int = 480  # 8 годин
```

Довгий час життя токена при відсутності refresh-токенів. Якщо токен скомпрометовано — він буде валідний 8 годин без можливості відкликання.

### 7.3 Відсутній механізм відкликання токенів

Немає blocklist/blacklist для JWT. Якщо потрібно примусово розлогінити користувача — це неможливо без зміни `SECRET_KEY` (що інвалідує *всі* токени).

## 8. Рекомендації за пріоритетом

### Високий пріоритет

1. **Додати тести** — мінімум для фінансових розрахунків (податки, ЄСВ, ВЗ, дохід)
2. **Налаштувати CORS** — замість `["*"]` вказати конкретні домени
3. **Вимагати SECRET_KEY** — не запускати сервер з дефолтним ключем
4. **Додати rate limiting** — на `/api/auth/login` та `/api/auth/register`

### Середній пріоритет

5. **Винести бізнес-логіку з routes** — створити сервісний шар
6. **Додати CI/CD** — мінімум lint + type check + тести
7. **Додати пагінацію** — для списків доходів/витрат
8. **Додати логування** — структуроване логування запитів і помилок

### Низький пріоритет

9. **Refresh-токени** — для безпечнішої роботи з сесіями
10. **Кешування звітів** — Redis або in-memory кеш для закритих періодів
11. **Репозиторний шар** — для кращої тестованості та повторного використання запитів
12. **Валідація паролів** — мінімальна довжина та складність

## 9. Висновок

Проект має **добру архітектурну основу**: правильний стек, асинхронність, типізація, чітка структура. Код читабельний, іменування зрозуміле, бізнес-домен (фінанси медичної ФОП) реалізований детально.

Основні проблеми стосуються **зрілості проекту**: відсутність тестів, CI/CD, логування, та деякі безпекові прогалини, типові для MVP-стадії. Найважливіше — додати тести для фінансових розрахунків, де помилка має пряму фінансову відповідальність.
